# Security-First Linting + Opt-in Tool Provisioning — Design

**Date:** 2026-06-03 · **Author:** ze (Zé) · **Status:** implementation

## Why

CSReview raises the security of code generated by AI agents. Its heuristic regex
detector is fast but, alone, produces false positives and misses what curated,
AST-based tools catch. To deliver a result **faithful to the objective — close to
certainty and far from false positives** — the coding agent must be able to use
the *right* tool per stack. That means CSReview must be able to **provision** the
security tools it needs.

Synthesis of three parallel analyses:
- **ChatGPT (positioning):** keep it honest — "local development-time security
  alignment (SAST/SCA, actionable reports, agent support)", **not** an automated
  pentest. Honest is more defensible and commercially stronger.
- **Qwen (Security-First Linting):** make stack-native security linters
  first-class, run with **security-only** profiles (no style noise), parse JSON,
  inject into the report.
- **Qwen (auto-install / "Trio de Ouro"):** prefer single-binary, fast,
  JSON-output tools; auto-provision them so the agent is self-sufficient.

## The tension and how we resolve it

Qwen's literal mechanism — silently installing linters **inside** the audited
project (`.csreview/env/`, mutating it) — conflicts with CSReview's hard
invariant: **read-only on the audited source; never install as a project
dependency; never touch the global system without consent.** We keep the *idea*
and make it safe:

- **Opt-in + informed consent.** Provisioning never runs silently. It is gated
  behind `--provision-tools` and prints exactly **what** will be installed,
  **from where** (official URL) and the **size** before any download. Default OFF.
- **Pinned official sources only.** A registry maps tool → official release host;
  the per-platform asset + version are resolved from the official releases API.
  No arbitrary/derived/redirected URLs.
- **Integrity.** The downloaded artifact's SHA-256 is verified against the
  vendor-published checksum **before** it is made executable or run. Mismatch →
  abort that tool, never execute.
- **Isolated, no pollution.** Binaries land in a CSReview-owned workspace
  (`.csreview/bin/`, gitignored), never in the user's `package.json`/global env,
  never with `sudo`/`-g`. A per-user cache makes repeat runs instant.
- **Fail-open cascade.** PATH → cache → (opt-in) download → isolated local
  install → degrade to "Agent-Only / lower confidence" with a clear report note.
  Never crashes, never blocks an offline scan.
- **Honesty.** The report states which tools actually ran vs heuristic-only, and
  lowers confidence when tools are absent. Tool + detector corroboration →
  `CONFIRMED` — this is the mechanism that **cuts false positives**.

## Architecture

- **`src/provision.js`** (foundation): pinned `TOOL_REGISTRY`
  (name → bin, official repo, asset templates, checksum source), `platformKey()`,
  `resolveLatestStableTag()`, `downloadArtifact()` + `verifyChecksum()` (sha256),
  `ensureTool()` implementing the fail-open cascade, path-safety for
  `.csreview/bin`. All network/exec/checksum operations are **injectable** so
  unit tests never touch the network.
- **Per-tool normalizers** (in `src/index.js`, mirroring `normalizeNpmAuditFindings`):
  Gitleaks, Bandit, gosec, Trivy → canonical `Finding` (`confidence:'TOOL-ONLY'`,
  `source:'<tool>'`). Reuse `deduplicateFindings`/`classifyToolMode`; Gitleaks
  corroborating the regex secret detector promotes to `CONFIRMED`.
- **Wiring** in `runAnalysis`: when a tool is available (PATH) or provisioned
  (opt-in), run it read-only with a **security-only** profile, write any temp
  config to `os.tmpdir()` (never the audited project), merge findings.
- **CLI**: `--provision-tools` flag + consent messaging + Agent-Only fallback.
- **Docs**: a README **landing** declaration (above the SKILL mirror) of the
  tools, the opt-in provisioning, and how to enable/decline; SKILL.md policy
  refined accordingly.

## Verified official sources (researched 2026-06-03)

Trio de Ouro: Semgrep (`semgrep/semgrep`, already orchestrated), OSV-Scanner
(`google/osv-scanner`, already orchestrated), **Gitleaks** (`gitleaks/gitleaks`).
Per-stack: **Bandit** (`PyCQA/bandit`), **gosec** (`securego/gosec`), **Trivy**
(`aquasecurity/trivy`, subsumes tfsec/checkov/hadolint), eslint-plugin-security
(`eslint-community/eslint-plugin-security`), brakeman (`presidentbeef/brakeman`),
ruff (`astral-sh/ruff`). IaC alternatives subsumed by Trivy: checkov
(`bridgecrewio/checkov`), hadolint (`hadolint/hadolint`), tflint
(`terraform-linters/tflint`).

### Research caveats (acted on)
- **local-php-security-checker** (`fabpot/local-php-security-checker`) is
  **archived** (read-only since Aug 2024) → prefer the maintained `composer
  audit` for PHP; archived binary only as a documented fallback.
- **eslint-plugin-security** warns (own README) it is **false-positive-prone** →
  weighted as **lower confidence / supplementary**, never auto-`CONFIRMED`.

## Implementation order (waves)

1. Provisioning foundation (`provision.js`) + tests — the security-critical core.
2. **Gitleaks** (secrets; biggest FP-reduction via corroboration).
3. **Bandit** + **gosec** (Python/Go AST, high signal).
4. **Trivy** (IaC/containers/FS).
5. Consent UX (`--provision-tools`) + report integration.
6. Docs (README landing + SKILL + mirror).
7. Adversarial review → remediate.

The remaining tools (eslint-plugin-security, composer audit, brakeman, ruff) stay
in the Stack-Native Recommendation Matrix and become straightforward registry +
normalizer additions on this foundation.

## Out of scope (honesty)
- No silent provisioning, no global/system installs, no writes to the audited
  source. No active probing of external/prod systems (unchanged invariant).
- Not all tools are engine-orchestrated in this PR; the rest are documented and
  the framework is ready.
